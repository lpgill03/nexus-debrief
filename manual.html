What's New in V98

V98.1 — RC/PDH Engine Lifecycle Fix (February 13, 2026)

Summary
Fixed a critical bug where toggling Ross Cameron or PDH profiles off/on in Mission Control did NOT restart the engine. The module-level singleton persisted across toggles, causing stale configurations to remain active.

Root Cause

Singleton Persistence: The _rc_engine and _pdh_engine module-level variables were never cleared by stop(). When users toggled off/on, get_ross_cameron_engine() or get_pdh_engine() returned the old instance with stale configuration.

Autopilot Disconnect: autopilot_service.stop(profile="RossCameron") set internal flags but never called the engine's stop() method, leaving threads running independently.

What Was Fixed

  • Singleton Clearing: Both ross_cameron/engine.py and pdh_hunter/engine.py now clear their module-level singleton (global _rc_engine; _rc_engine = None) in the stop() method
  • Autopilot Integration: New _stop_profile_engine() method in autopilot_service.py now calls engine.stop() for RC and PDH profiles
  • Fresh Instance on Restart: Next get_ross_cameron_engine() or get_pdh_engine() call creates a fresh instance that re-reads config from disk

Files Modified

File Changes
nexus_investing/services/ross_cameron/engine.py stop() clears singleton via global _rc_engine; _rc_engine = None
nexus_investing/services/pdh_hunter/engine.py stop() clears singleton via global _pdh_engine; _pdh_engine = None
nexus_investing/services/autopilot_service.py Added _stop_profile_engine(profile), called from stop(profile=...)

V98.0 — PDH Cash-Governed Entry + Drop-to-Upgrade + Council Priority (February 13, 2026)

Summary
Replaced hardcoded 5-position limit with exposure-based entry gating. Positions are now governed by cash (25% base / 35% extended), not an arbitrary count. When exposure cap is hit and a strong pick appears, the engine either drops the weakest dying position or extends into the 35% zone if all positions are healthy.

Design Philosophy

The engine is a Python process, not a human — it can track N positions simultaneously. The real constraint is cash and risk, not a slot count. The hardcoded max_concurrent_positions: 5 was removed as a gate and replaced with exposure percentage math. A safety backstop at 15 positions remains to prevent runaway bugs.

Three Core Rules (V98.0)

Rule 1: Council Priority Ordering

Premarket council conviction picks (conviction >= 70 on 0-100 scale, go=True) are processed first in the detection loop. Non-conviction candidates retain original relative order. This gives high-conviction picks first chance at fast-track entry and slot allocation.

Rule 2: Exposure-Based Gating

Total exposure < 25% → enter freely (no position count limit). Total exposure >= 25% → composite score >= 85 required to proceed to Rule 3.

Rule 3: Drop-to-Upgrade Logic

When a strong pick (composite >= 85) appears and exposure is >= 25%, the engine decides:

  • Drop Found: Position with composite < 40 AND negative P&L → force-exit via market sell, enter new position
  • Extended Zone: Nothing to drop but exposure < 35% → extend into 35% zone, enter new position
  • No Room: Nothing to drop AND extended cap (35%) reached → skip entry

Exposure Gating Logic

total_exposure < 25% → enter freely (Rule 2)
total_exposure >= 25% AND < 35%:
  → composite >= 85?
    → weak position found (composite < 40, negative P&L)? → DROP it, enter (Rule 3a)
    → all healthy? → extend to 35% zone, enter (Rule 3b)
    → composite < 85? → skip entry
total_exposure >= 35% → hard block

What Changed

  • _check_safety_rails() rewritten — Now returns (bool, reason) tuple. Reasons include: exposure_cap, extended_exposure_cap, daily_loss_limit, max_trades, time_cutoff, position_backstop, already_in_position
  • _execute_signals() updated — Uses structured safety rail response. On exposure_cap, tries _try_drop_to_upgrade()
  • _detect_pumps() updated — Candidates reordered via _apply_council_priority() before iteration
  • Config: max_concurrent_positions changed from 5 → 15 (safety backstop only)

What Was Added

  • _try_drop_to_upgrade(new_composite, new_symbol) — V98.0 Rule 3 implementation
  • _force_exit_position(symbol, reason) — Market-sells a position immediately for drop-to-upgrade
  • _apply_council_priority(candidates) — V98.0 Rule 1 implementation for conviction-based ordering
  • _get_account_equity() — Extracted helper (returns account equity or $100K default)
  • _get_total_exposure() — Extracted helper (sums shares * entry_price across positions)
  • Config keys: extended_exposure_pct: 35.0, drop_to_upgrade_min_composite: 85, drop_victim_max_composite: 40

Verified

  • TestRule3_DropToUpgrade (7 tests): drop weakest, extend zone, extended cap reached, composite threshold, multiple victims, positive P&L not dropped, won't drop self

Chapter 1: What is Nexus Investing?

1.1 Overview

Nexus Investing is an autonomous AI-powered trading system that combines specialized artificial intelligence agents into a "council" that debates and decides on stock trades. Think of it like having a team of expert analysts—a fundamentals expert, a technical chart reader, a news analyst, a risk manager, and others—all working together 24/7 to manage your portfolio.

Typical Trading App Nexus Investing
Shows you data, you decide AI council debates and decides
One algorithm 8 specialized AI agents
One-size-fits-all 6 risk profiles (Conservative to PDH)
Manual trading Fully autonomous execution
No learning Learns from every trade
Retail-level tools Institutional-grade risk controls

1.2 The Multi-Engine Architecture (6 Profiles)

Nexus operates like a professional hedge fund with multiple trading approaches:

NEXUS INVESTING V98.0 — 6-PROFILE ARCHITECTURE

FAST LANE (Python-only, sub-second decisions):
  Aggressive     — Momentum breakouts, 5 positions max
  APEX Elite     — Multi-strategy rotation (3 active), Python gatekeeper
  PumpDumpHunter — Micro-cap pump detection, V98 exposure gating
  RossCameron    — Small-cap momentum, first 60 min only

STRATEGIC LANE (LLM Council, minutes per decision):
  Conservative   — Capital preservation, dividend aristocrats
  Moderate       — Balanced growth, quality-momentum

1.3 The AI Council

When analyzing a stock, 8 specialized AI agents each contribute their expertise:

Agent Role What They Look At
Market Analyst Price & Volume Current price, volume spikes, moving averages
Fundamentals Analyst Company Health Revenue, profit, debt, valuation ratios
News & Sentiment Current Events News articles, social sentiment, analyst ratings
Calendar Agent Upcoming Events Earnings dates, dividends, FDA approvals
Bull Researcher Makes the Case FOR Why this stock could go up
Bear Researcher Makes the Case AGAINST Why this stock could go down
Risk Manager Danger Assessment What could go wrong, position sizing
Trader Agent Final Decision Synthesizes all inputs, makes BUY/SELL/HOLD

How Decisions Are Made

  1. All agents analyze the stock simultaneously
  2. Bull and Bear researchers debate
  3. Risk Manager flags any concerns
  4. Trader Agent weighs all inputs
  5. A confidence percentage (0-100%) determines action
  6. Trades only execute if confidence exceeds profile threshold

3.3 Circuit Breaker

The Emergency OFF Switch

If anything goes wrong, the circuit breaker immediately:

  • Stops all new trades
  • Prevents any orders from being placed
  • Keeps existing positions (does not auto-sell)
  • Requires manual re-activation

How to Activate

  1. Dashboard → Settings → Circuit Breaker → OFF
  2. Or set CIRCUIT_BREAKER=false in environment

Chapter 5: Troubleshooting

5.1 Common Issues

"No trades happening"

  • Check circuit breaker is ON
  • Check if in midday pause (11 AM - 2 PM)
  • Check if cash is available
  • Check confidence thresholds (may be too high)
  • In DEV mode, thresholds are lowered to see more action

"Too many trades"

  • Increase confidence threshold
  • Switch to PROD mode
  • Reduce position limits

"System says HOLD on everything"

  • May indicate market-wide uncertainty
  • Check VIX level (high = fewer opportunities)
  • Review quality screen results
  • Normal during crisis regimes

"API errors"

  • Check API keys in Settings
  • Verify Alpaca account status
  • Check FMP API limits

5.2 Getting Help

  1. Check this manual first
  2. Review the CHANGELOG for recent changes
  3. Export dashboard logs for analysis
  4. For technical issues, consult Technical Reference (Part 2)
